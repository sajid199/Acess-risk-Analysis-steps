Section 2

Risk Types in SAP GRC ARA

Key Concept: Action = T-Code  |  Permission = Authorization Values
SoD

Segregation of Duties Risk

Combination of conflicting functions. Occurs when duties that should be separated are assigned to one user.

Examples SU01 + PFCG   SCC4 + STMS
SoD Risk Levels (Action Level):
  • T1 Display + T2 Display
  • T1 Full + T2 Display
  • T1 Display + T2 Full
  • T1 Full + T2 Full → Highest Risk
CA

Critical Action Risk

A single transaction that is itself sensitive or dangerous.

Critical T-Codes SCC4 SCC5 SE38 SPRO STMS
CP

Critical Permission Risk

Sensitive authorization object + field + value combinations.

Examples

SCC4: S_TABU_DIS with Activity 01 or 02

SE38: S_DEVELOP with Activity 01 or 02

Why Is This a Risk? (BENZ_USER1 Scenario)

Using SU01 — the user can:

  • Create / change users
  • Reset passwords
  • Assign powerful roles (e.g., SAP_ALL)
+

Using SCC4 — the user can:

  • Change client settings
  • Modify protection levels
  • Change client control parameters
=

⚠️ SoD Violation

One person can create a powerful user, assign sensitive roles, change client settings, and bypass controls — all without anyone else's approval.

How GRC Detects It: GRC ARA checks — Does the user have access to BENZ_SEC AND BENZ_CA? If Yes → Risk Found: BNZ_RSK1

Section 7

Interview Questions & Answers

Common SAP GRC ARA interview questions with clear, interviewer-friendly explanations.

Q1. Are you using Global Rule Sets or Custom Rule Sets in your company?
Answer: "In our company we use both. We started with the SAP Global Rule Set as a baseline — it covers standard risks across Finance, HR, and other processes based on industry best practices. However, every company has unique processes, so we created custom rule sets specific to our business. For example, we created custom Business Processes, Functions, and Risk IDs to match our internal T-code landscape. The combination of Global and Custom rule sets gives us comprehensive coverage."
Q2. Have you ever created a custom risk?
Answer: "Yes. We created a custom SoD risk called BNZ_RSK1 which combines two functions — BENZ_SEC (Security Administration: SU01, PFCG, PFUD) and BENZ_CA (Client Administration: SCC4, SCC5, SCCL, SCC9). The reason is that if one person can both manage users and change client settings, they can create a backdoor user and bypass security controls. GRC ARA detects this by checking if a user has access to both functions simultaneously."
Q3. What is the difference between Action Level and Permission Level risk analysis?
Answer: "Action Level analysis checks at the T-Code level — whether a user has the listed transactions. Permission Level goes deeper and checks authorization objects, fields, and values. For example, at action level we check if a user has both SU01 and SCC4. At permission level we verify the exact authorization objects like S_USER_GRP (Activity 01/02) and S_TABU_DIS (Activity 01/02, Auth Group SS). Permission level is more accurate but heavier on system resources."
Q4. What is an SoD Risk? Give a real example.
Answer: "SoD stands for Segregation of Duties. A risk occurs when one person has access to two conflicting activities that together can be misused. For example, in our BASIS environment, the Security Admin role (SU01, PFCG) and the Client Admin role (SCC4, SCC5) should belong to different people. If one person has both, they could create an unauthorized superuser, assign SAP_ALL, and then change client protection — all without any oversight. This is a critical SoD violation."
Q5. What is the difference between Critical Action and Critical Permission risk?
Answer: "Critical Action Risk is about a single dangerous T-Code — for example SCC4 (Change Client Settings) or SE38 (ABAP Editor) are powerful on their own, even without being combined. Critical Permission Risk is about dangerous authorization object/value combinations — for example S_DEVELOP with Activity 01 or 02 gives ABAP development access which is highly sensitive. Critical risks use one function (unlike SoD which uses two), because the risk exists in the single capability itself."
Q6. What T-codes do you use for monitoring in SAP GRC?
Answer: "For monitoring sync jobs, I use SM37 (Job Monitor) to check if background jobs ran successfully. If there are errors, I use SLG1 (Application Log) to view detailed error logs and identify what failed during the sync. For GRC configuration I primarily work through NWBC (NetWeaver Business Client) and SPRO."
Q7. What happens if you forget to Generate the Risk ID after creating it?
Answer: "If you don't generate the Risk ID, the risk rules are not activated in the system. GRC ARA will not detect any violations for that risk during analysis. You must always select the Risk ID and click Generate (Foreground for small sets, Background for large). The system generates individual Rule IDs — one for each T-code combination in the functions. Only after generation does the risk become active and detectable."
Q8. How does GRC ARA identify an SoD violation?
Answer: "GRC ARA compares user role assignments against the defined risk rules. For an SoD risk like BNZ_RSK1 = BENZ_SEC + BENZ_CA, GRC checks: Does this user have access to Function 1 (BENZ_SEC)? AND Does this user have access to Function 2 (BENZ_CA)? If both conditions are true, it flags a violation. The risk analysis can run at User, Role, Profile, or HR Object level depending on the scope needed."
